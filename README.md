# 文本数据清洗工具（Text Data Cleaner）

## 项目简介

本项目是一个基于 Python 3.10+ 和标准库实现的文本数据清洗工具，主要用于对 `.txt` 文本进行基础预处理。工具支持去除空行、去除重复行、忽略大小写去重、自定义裁剪字符、输出统计报告、指定输出文件，以及 UTF-8 / GBK 编码兼容读取等功能。

项目强调“小步迭代开发”，每一轮只实现一个明确的小目标，并在每轮结束后进行测试、提交与过程记录，便于完整追踪 AI 辅助开发过程。

## 选题说明

本项目的选题来源于一次面向岗位要求的 AI 辅助分析过程。

最初，我将岗位需求、能力要求等信息提供给 GPT，要求它从多个不同方向生成若干可选题目。这些选题从实用性、可实现性、复杂度、展示效果等不同角度展开，最终从中筛选出了一个更适合作为阶段性开发考核项目的主题：**简单、实用、便于迭代扩展的文本数据清洗工具**。

选择该题目的原因主要有以下几点：

- 文本清洗属于高频、实用的问题场景
- 功能可以天然拆分为多轮逐步开发
- 项目复杂度适中，适合展示 AI 辅助开发流程
- 便于记录每一轮功能演进和测试过程

## 功能特性

当前项目已实现的主要功能包括：

- 自动读取文本文件
- 优先使用 UTF-8 读取，失败后自动尝试 GBK
- 去除空行
- 去除重复行
- 支持 `--ignore-case` 忽略大小写去重
- 支持 `--strip-chars` 去除每行首尾指定字符
- 支持 `-o` 指定输出文件路径
- 未指定输出文件时，自动生成 `原文件名_cleaned.txt`
- 输出清洗统计报告
- 对空文件、异常编码、缺失文件等边界情况进行处理
- 采用逐行迭代方式处理，避免一次性 `readlines()` 导致大文件内存占用过高

## 运行环境

- Python 3.10+
- 仅使用 Python 标准库
- 无需安装第三方依赖

## 启动方式

进入项目目录后，直接运行：

```powershell
cd D:\Python\text-data-cleaner
python clean.py test_input.txt
```

如果希望指定输出文件：

```powershell
python clean.py -o output.txt test_input.txt
```

如果希望忽略大小写去重：

```powershell
python clean.py --ignore-case test_input.txt
```

如果希望去除每行首尾的指定字符：

```powershell
python clean.py '--strip-chars=",' test_input.txt
```

多个参数可以组合使用：

```powershell
python clean.py --ignore-case '--strip-chars=",' -o output.txt test_input.txt
```

## 输出结果说明

程序运行完成后会有两部分输出：

1. 清洗后的文本内容会写入输出文件
2. 控制台会打印清洗统计报告

统计报告包括：

- 原始总行数
- 清洗后行数
- 去除空行数
- 去除重复行数

## 提示词产生方法

本项目的提示词不是一次性生成全部内容，而是先对功能进行拆解，再逐轮设计。

具体方法如下：

1. 先把工具需要具备的功能逐项列出
2. 要求 GPT 将完整开发任务拆分为多轮完成，而不是一次生成全部功能
3. 将每一项功能的目标、约束和说明继续发送给 GPT
4. 要求 GPT 完善并生成适合每一轮开发的提示词

通过这种方式，可以让提示词更聚焦当前轮次的小目标，同时便于控制开发节奏，也方便记录每一轮的输入、修改和输出结果。

## 开发过程说明

本项目采用逐轮迭代开发方式进行，每轮只完成一个小功能，并严格执行：

- 修改代码
- 运行测试
- Git 提交
- 推送到 GitHub
- 追加 JSONL 过程记录

整个开发过程中，每一轮都围绕新增功能进行验证。例如在新增某项功能后，会创建或调整测试文本文件，专门验证这一项功能是否按预期生效，同时确保旧功能不被破坏。

这种开发方式使项目具备以下优点：

- 每次修改范围清晰
- 错误定位更容易
- 便于展示 AI 辅助开发过程
- 适合考核或复盘分析

## JSONL 文件生成方法

为了完整记录 AI 辅助开发过程，在项目根目录下的 `AGENTS.md` 中，明确要求 Codex 在每一轮实现功能后：

1. 完成代码测试
2. 执行 Git 提交
3. 推送到 GitHub
4. 在 `AI开发考核_梁嘉俊_文本数据清洗工具.jsonl` 中追加一行 JSON 数据

JSONL 中每条记录包含以下字段：

- `round_id`
- `prompt_content`
- `modify_diff`
- `commit_hash`
- `modify_time`
- `agent_type`
- `dev_language`

其中：

- `prompt_content` 用于记录每一轮的自然语言提示词
- `modify_diff` 用于保存本轮提交对应的 Unified Diff
- `commit_hash` 用于关联具体版本
- `modify_time` 用于记录该轮完成时间

这种方式可以将“提示词 -> 修改内容 -> 提交记录 -> 时间信息”串联起来，形成完整、可追踪的开发链路。

## 过程中遇到的问题与解决方法

在整个开发过程中，最频繁出现的问题主要集中在 **推送到 GitHub** 这一步。

### 主要问题

在每一轮对话完成后，Codex 在执行 `git push` 时经常因为网络问题失败，典型表现包括：

- 连接被重置
- 无法连接到 GitHub 443 端口
- 需要手动重新执行推送

### 解决方法

- 发现推送失败后，优先进行重试
- 必要时手动再次执行 `git push`
- 在过程中持续保留本地提交，避免因网络问题导致代码丢失
- 将问题与报错信息明确记录，便于排查

也就是说，代码开发和本地提交通常是正常完成的，主要不稳定点在于远程网络连接，这也是 AI 自动化开发流程在真实环境中常见的一个实际问题。
